## 🎥 WebRTC fundamentals

### The three-layer model

- **Signaling** — introductions between peers. Any transport (WebSocket, HTTP, SSE, carrier pigeon). Relays SDP + ICE candidates. Never sees media. Cheap.
- **Media** — encrypted audio/video flows peer-to-peer over UDP once ICE picks a working path. Never touches your server.
- **NAT traversal** — the tools each peer uses to figure out how it can be reached: STUN, TURN, ICE.

### STUN vs TURN vs ICE

- **STUN** — one-packet exchange with a public server. Reply: "I saw you arrive from `<public-IP>:<port>`." Cheap, ambient. Use always.
- **TURN** — relay server that forwards media when direct P2P fails. Costs real money (bandwidth). Handles ~20% of real-world users behind symmetric NAT / corporate firewalls.
- **ICE** — the algorithm inside each browser. Gathers candidates (host, srflx, relay), exchanges them via signaling, tries every pair via STUN pings, picks the winner.

### SDP offer/answer

- **Offer** — SDP blob generated by the calling peer: "here's what I want to send/receive, my codecs, my keys."
- **Answer** — SDP blob from the receiving peer: "here's what I'll accept."
- Both sides `setLocalDescription`/`setRemoteDescription` to install them. The SDP transits your signaling channel opaquely.

### Perfect negotiation (polite/impolite)

- Handles **glare** — when both peers try to send an offer simultaneously.
- One peer is "impolite" (initiates the offer, wins collisions).
- Other is "polite" (yields, rolls back its local description).
- Convention we used: newer joiner = impolite; older = polite.
- Without this, a real symmetric collision breaks the call.

### Data channels

- Same `RTCPeerConnection` can carry audio, video, and arbitrary data.
- Data channels ride over **SCTP over DTLS** (encrypted, ordered or unordered).
- We use one for **live captions** — text of what one peer says, transmitted to the other peer's browser.

### Encryption is always on

- **DTLS** handshake derives keys.
- **SRTP** encrypts every RTP audio packet.
- You can't turn it off. Even the ICE STUN pings are integrity-protected.
- In P2P: truly end-to-end encrypted (no one but the two peers has the keys). In SFU: media is encrypted in transit, but the SFU has keys.

## 🏗️ Architecture patterns

### P2P vs SFU vs MCU

| | P2P (mesh) | SFU | MCU |
|---|---|---|---|
| Server cost | $0 | Medium | High |
| Best for | 2–4 people | 3–50+ | Legacy conf. |
| Media E2E encrypted | Yes | No (server sees keys) | No |

DuoCall is P2P. Zoom, Meet, Discord all use SFU for group calls.

### Ownership boundaries (the DuoCall example)

- **`SignalingClient`** — owns the WebSocket. Talks to the server.
- **`CallController`** — owns the `RTCPeerConnection`. Talks to the peer.
- **`MicProvider`** — owns the `MediaStream`. App-lifetime resource.
- Clean boundaries mean each class is testable in isolation and replaceable.

### Signaling protocol shape

- Discriminated union of JSON messages: `join`, `resume`, `leave`, `signal`, `ping` (client → server); `joined`, `resumed`, `peer-joined`, `peer-left`, `signal`, `error`, `pong` (server → client).
- Server-side switch dispatches to a `handleXxx()` per message type. Each handler closes over `clientId`/`send`/`rooms` — no parameter clutter.

## 🖥️ Server / signaling

### Room ID security

- **Server-generated** `nanoid(21)` → ~149 bits of entropy → non-guessable.
- Server stores each ID in a `Map`, so `join` validates the room exists (prevents brute-force squatting).
- Room creation rate-limited per IP (`@fastify/rate-limit`).

### Session tokens + graceful reconnect

- Server issues a `nanoid(32)` token on `join`. Client keeps it in memory.
- On WebSocket drop, the server holds the slot for a **15-second grace period**.
- Client reconnects with the token → server reattaches to the same `clientId` → the peer never sees a disconnect.
- This is the Slack/Discord/Meet pattern.

### App-level heartbeats

- Client pings every **25 seconds**. Server responds with pong.
- Proxies (Fly, Cloudflare, most CDNs) idle-close WebSockets around 60s. Pings keep the pipe warm.
- Server sweeps stale clients that haven't heartbeat within 60s.

### In-memory state trade-off

- `Map<roomId, Set<Client>>` in the Node process. Fast, simple, no DB.
- Trade-off: rooms don't survive a server restart. Also blocks horizontal scaling.
- Production would move room state + rate limits into **Redis with pub/sub** so any signaling node can serve any room.

### Fastify vs Express

- **Fastify** (modern): async-native, first-party `@fastify/websocket` plugin, structured JSON logs via `pino`, schema-driven validation, faster routing.
- **Express** (older): callback middleware, no async/await native support, no built-in JSON validation, larger third-party ecosystem.
- Fastify is the modern default for small-to-mid JSON+WebSocket services.

## 🌐 Chrome quirks

### Regular vs incognito can't connect via WebRTC

- Chrome hides real LAN IPs by replacing host candidates with mDNS names like `abc.local`.
- **Chrome intentionally prevents regular and incognito profiles from resolving each other's mDNS names** — that's a core privacy feature of incognito.
- Result: WebRTC connection can't complete between them. Signaling works fine; only ICE fails.
- **Chrome + Safari works** because Safari's mDNS resolver is independent of Chrome's rules.
- Only fix: TURN (relay bypasses direct P2P entirely).

### Microphone permission states

- **`prompt`** — user hasn't answered / dismissed the popup. Calling `getUserMedia()` re-shows the popup.
- **`granted`** — user allowed. `getUserMedia()` returns instantly.
- **`denied`** — user clicked Block. **No JS can force the popup to reappear.** User must reset via lock icon → Microphone → Allow.
- Detect via `navigator.permissions.query({ name: "microphone" })`.

### SpeechRecognition

- `window.SpeechRecognition` / `window.webkitSpeechRecognition`. Chrome + Edge + modern Safari. **Not** Firefox.
- Uses cloud service — needs internet.
- Its `.abort()` releases the mic immediately; `.stop()` is a graceful async shutdown that can take hundreds of ms.

## 🧩 React / frontend patterns

### Global provider for shared resources

- The **mic is app-lifetime**, not per-call. Acquire once via `MicProvider`, share via `useMic()`.
- Same pattern for any expensive resource that outlives individual pages (camera, WebSocket, WebGL context).
- Trying to acquire per-page creates cumulative session churn and StrictMode races.

### StrictMode double-mount gotchas

- Dev-only: React mounts → cleans up → mounts again to catch cleanup bugs.
- If a `useEffect` starts an async operation (`getUserMedia`, `fetch`, WebSocket) and cleanup runs before it resolves, the resulting resource can leak.
- Fix: create instances **inside** the `useEffect`, not via `useMemo`. Cleanup then closes them; new instance on next mount.

### Custom hook dedup pattern

- `MicProvider` uses `useRef` for an in-flight promise. Multiple concurrent `ensureMic()` callers all get the same promise → one `getUserMedia` call.
- Standard pattern for de-duping concurrent async work in React.

### Perfect-negotiation state on the client

- Track `isPolite`, `makingOffer`, `ignoreOffer` as `RTCPeerConnection`-adjacent state.
- Handle `signalingState !== "stable"` as a collision indicator.

## 🧬 TypeScript idioms

### Const-object-as-const instead of `enum`

```ts
export const CallState = {
  Idle: "idle",
  Connected: "connected",
} as const;
export type CallState = (typeof CallState)[keyof typeof CallState];
```

- **Why not `enum`?** Generates runtime code (bidirectional lookup), has `isolatedModules` incompatibility issues (Vite requires it), weaker literal-narrowing.
- **This pattern gives you**: enum-like usage (`CallState.Connected`), literal string types (not `string`), zero runtime footprint beyond the object itself.
- Modern TypeScript best practice.

### Discriminated union messages

```ts
type ClientMessage =
  | { type: "join"; roomId: string }
  | { type: "signal"; to: string; payload: SignalPayload };
```

- Switch on `msg.type` and TypeScript narrows automatically.
- `Extract<ClientMessage, { type: "join" }>` pulls out one variant for helper functions.

### Testable pure functions

- Instead of testing a top-level constant computed from `import.meta.env`, extract a `deriveServerUrls(env, location)` function and test that. Zero mocking needed.
- Applied to `client/src/env.ts` and `server/src/config.ts`.

## ✅ Testing patterns

### Vitest setup for a Vite project

- `vite.config.ts` gains a `test` block — same tool, no separate Jest config.
- `jsdom` environment for React tests; `node` for server tests.
- `setupFiles` for `@testing-library/jest-dom/vitest` and manual `afterEach(cleanup)` when `globals: false`.

### Mocking WebRTC

- `RTCPeerConnection` and `getUserMedia` need small classes that mock the event/method surface: `addTrack`, `createOffer`, `createAnswer`, `addEventListener`, etc.
- `MediaStream` and `MediaStreamTrack` need minimal shims.
- Data channel: mock `createDataChannel` to return a fake with `readyState`, `send`, `close`, `addEventListener`.

### Fake timers for time-sensitive behavior

- `vi.useFakeTimers()` + `vi.advanceTimersByTime(ms)` for testing ping intervals, reconnect backoff, grace-period eviction.
- `vi.advanceTimersByTimeAsync(ms)` when the tested code awaits between timer fires.

### Flushing microtasks

- Sometimes chained awaits need multiple microtask cycles to complete. Helper: `for (let i = 0; i < 20; i++) await Promise.resolve();`
- Cleaner than counting exact microtask counts.

## 🚀 Deployment on Fly.io

### The mental model

- `fly launch` once (creates the app + writes app name into `fly.toml`).
- `fly deploy` every time thereafter (builds Dockerfile remotely, does rolling deploy).
- `fly releases` shows version history; `fly releases rollback` is near-instant.

### Docker multi-stage build

- Stage 1: build client (Vite → `dist/`).
- Stage 2: build server (tsc → `dist/`).
- Stage 3: runtime — `npm ci --omit=dev`, copy both `dist/`s, `CMD ["node", "dist/index.js"]`.
- Prod Fastify serves both the API and the built React app on the same origin.

### Same-origin in production

- Prod client's `env.ts` uses `window.location.origin`; dev falls back to `localhost:8080`.
- Same-origin eliminates CORS entirely in prod.

## 📈 Scaling and cost

### 10k rooms/day capacity

| Component | Fine? | If not, fix |
|---|---|---|
| STUN | ✓ free | — |
| Signaling | ✓ one node | Redis-shared rooms + pub/sub |
| In-memory rooms | ✗ | Move to Redis |
| Rate limit counters | ✗ | Move to Redis |
| TURN | 💰 real money | See below |

### Cost math

- ~20% of calls need TURN.
- At 10k rooms/day × 10 min × 128 kbps × 20% ≈ 2.6 TB/month.
- At $0.40–$1/GB (managed TURN) ≈ **$1k–$2.5k/mo**.
- Everything else (STUN, signaling, storage) is negligible next to this.

### TURN "in real life"

- Ship with **three transports** in the ICE server list: UDP, TCP, TLS-on-443.
- TLS-on-443 is indistinguishable from HTTPS — survives deep-packet-inspecting corporate firewalls.
- Providers: Twilio, Cloudflare Calls, Xirsys, Metered. Or self-host `coturn` on cheap VMs.

## 🧠 System design mental models

### The one-sentence WebRTC summary

> WebRTC gives two browsers a way to talk directly over encrypted UDP; STUN discovers the addresses they can reach each other at, TURN relays when they can't, ICE decides which path works, and signaling — which you build yourself, typically over WebSocket — is how they exchange the addresses and codecs in the first place.

### "What breaks at scale?" pattern

- The **thing you didn't measure**. In DuoCall's case: in-memory state breaks the moment you go to 2+ nodes.
- The **thing that scales linearly with users**. Here: TURN egress.
- The **thing that shows up on someone else's bill**. Here: also TURN.

### Framework choice heuristic

- Is the app a **collection of pages** (marketing, blog, dashboard with SEO)? → Next.js.
- Is the app a **single interactive tool** (like DuoCall)? → Vite + React.
- Is real-time signaling involved? → separate Node/Fastify server, never Next.js serverless routes.

## 🛠️ Meta / process learnings

### Iterative planning with Claude

- `PLAN.md` as a **living design document**. Iterate before coding. Cheap to change; expensive to change later.
- Ask "why" for each decision. Force yourself (or Claude) to justify.
- Convert open questions into decisions as they get resolved.

### `DECISIONS.md` for defensibility

- Not implementation detail. Not repeated in code.
- Only tradeoffs, scaling analysis, and reasoning that a reviewer or future-you would appreciate.
- Ordered by impact (highest-first), not chronological.

### `CLAUDE.md` for continuity

- Repo-scoped context that every future Claude Code session picks up automatically.
- Directory layout, non-obvious conventions, known gotchas, commands cheat sheet.
- Save future-you 15 minutes of re-familiarizing.

### Commit hygiene

- Commit after each feature/refactor, not batched. Imperative messages.
- Push regularly. Don't lose work.
- Small commits make history readable and revertible.
